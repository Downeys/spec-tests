## Typography

Four sizes, two weights. Tailwind v4 default scale.

| Role | Tailwind | Size | Weight | Line Height | Usage in Phase 2 |
|------|----------|------|--------|-------------|------------------|
| Body | `text-sm` | 14px | `font-normal` (400) | `leading-6` (1.71) | Chat messages (user + assistant), composer input, wiki-citation body, tool-trace summary lines |
| Label | `text-xs` | 12px | `font-medium` (500) | `leading-4` (1.33) | Tool-trace row meta (`tavily_extract`, `→ 4823 chars`), status pill ("Last compiled: 14:32 • 3 claims unwritten"), claim-ID inline citations (`[[claim:01J9X…]]`), confidence/status badge text |
| Heading | `text-lg` | 18px | `font-medium` (500) | `leading-7` (1.55) | Empty-state heading; recompile post-success inline-system message ("Recompiled: 1 page written"); assistant message group label if rendered |
| Display | `text-2xl` | 24px | `font-medium` (500) | `leading-8` (1.33) | App-shell title in header bar ("Business Strategy Planner") |

**Two weights, no exceptions:** `font-normal` (400) for prose; `font-medium` (500) for headings, labels, and any element that must read as structural. Do not introduce `font-semibold` or `font-bold` in Phase 2 — `font-medium` reads as emphasis at this size scale and against the off-white surfaces.

**Body line-height = 1.71** (Tailwind's `leading-6` against `text-sm`) is intentionally generous: chat messages mix prose with embedded ULID citations and confidence numbers, and the extra line-height keeps the scan readable. Heading line-height = 1.55 (`leading-7` against `text-lg`) is tighter to keep heading mass coherent.

**Numerics in chat (confidences, ULIDs, dollar amounts) are rendered in the same `font-sans` body face — not a tabular monospace.** Rationale: D-09's hypothesis framing reads as prose ("One hypothesis we have — confidence 0.55 — is that…"); switching to `font-mono` mid-sentence would fight that prose rhythm. ULIDs in the inline `[[claim:01J9X…]]` form are short enough to read inline in sans. Tool-trace expanded view (where alignment matters) is the one place `font-mono` may appear — see Tool Trace section in Component Inventory below.

## Color

Phase 2 follows the 60/30/10 rule via shadcn's CSS-variable token system (Tailwind v4's `@theme` block writes the same tokens). Light mode is the v1 default; dark mode is wired by shadcn but not a Phase 2 polish target. Concrete hex values are recorded here so the auditor can verify against rendered output.

| Role | Token (Tailwind v4 @theme) | Hex (light) | Hex (dark) | Usage |
|------|---------------------------|-------------|------------|-------|
| Dominant (60%) | `--color-background` / `bg-background`, `--color-foreground` / `text-foreground` | `#FFFFFF` / `#0A0A0A` | `#0A0A0A` / `#FAFAFA` | App-shell background; chat viewport background; default text color |
| Secondary (30%) | `--color-muted` / `bg-muted`, `--color-muted-foreground` / `text-muted-foreground`, `--color-border` / `border-border`, `--color-card` / `bg-card` | `bg-muted #F4F4F5`, `border-border #E4E4E7`, `text-muted-foreground #71717A` | `bg-muted #27272A`, `border-border #27272A`, `text-muted-foreground #A1A1AA` | Header bar background; composer background; tool-trace expanded-row background; wiki-citation block background; user-message bubble fill; status-pill idle fill; the "card" surfaces around interactive content |
| Accent (10%) | `--color-primary` / `bg-primary`, `--color-primary-foreground` / `text-primary-foreground` | `#18181B` / `#FAFAFA` | `#FAFAFA` / `#18181B` | **Reserved-for list below — never used outside it** |
| Destructive | `--color-destructive` / `bg-destructive`, `--color-destructive-foreground` / `text-destructive-foreground` | `#DC2626` / `#FAFAFA` | `#7F1D1D` / `#FAFAFA` | SSE-connection-lost banner; sub-agent schema-failure structured-error chat message; quantitative-claim-guard rejection message |

**Accent reserved for** (this is the explicit reserved-for list — never broadened):

1. **Recompile button (idle)** — `bg-primary text-primary-foreground` on the header-bar primary action. The whole button, not a hover-only state.
2. **Composer send button** — `bg-primary text-primary-foreground` on the assistant-ui `ComposerPrimitive.Send` (the round arrow-up button). Disabled state drops opacity to 30% (`disabled:opacity-30`) per assistant-ui idiom.
3. **"Open in Obsidian" button** inside a wiki-citation block (D-14) — same `bg-primary text-primary-foreground` treatment so the deeplink reads as the call-to-action of the citation.
4. **Inline `[[claim:01J9X…]]` citation links** — `text-primary` underline-on-hover, no background fill. They are the prose equivalent of the recompile/send buttons: the system's *evidence pointers*, the most semantically loaded element in chat.

That's it. Four uses. Hover states are `hover:bg-primary/90` (Tailwind opacity modifier, no separate hover token). **Do not** apply `bg-primary` to: tool-trace row backgrounds, status-pill (idle), user-message bubbles, hover affordances on links other than claim citations, or any "this is interactive" affordance. Interactive elements use the secondary palette (border + bg-muted + text-foreground); accent is reserved for *evidence-bearing* and *primary-action* elements only.

**Destructive** is reserved for three error states (Phase 2 explicit list):

1. SSE disconnect mid-stream — banner inside the chat viewport ("Connection lost — partial response saved. Reconnect?")
2. Sub-agent schema-malformed terminal failure (eval dimension #5) — inline system message in chat naming the failed Zod path
3. Quantitative-claim-guard rejection — inline system message naming the rule (handled as a *system* message, not as a CRIT-01 pushback; CRIT-01 pushback is a normal-tone assistant message, not destructive-color)

The CRIT-01 pushback (D-07) is a **normal-tone** assistant message — `text-foreground` on `bg-muted` user-message-style. Pushback is part of the conversation, not an error. Destructive color is reserved for actual system failures.

**Confidence rendering (D-09 hypothesis framing).** Phase 2 renders confidence as **prose plus a label-styled inline span** — never a colored badge. Format: `"confidence 0.55"` rendered as a `text-xs text-muted-foreground` inline span. Phase 5's UI-05 confidence badges add color encoding; D-09 explicitly states "the inline visual confidence badge (UI-05) is added in Phase 5 for reinforcement, not as a substitute" — so Phase 2 must not preempt that with green/yellow/red coloring. Honor the phase boundary.

## Component Inventory

The Phase 2 chat surface is six concrete components. Each is listed with its assistant-ui parent primitive (or "custom"), its source location, the locked decisions it implements, and the design contract it must honor.

### 1. App shell — `src/ui/App.tsx`

- **Implements:** UI-01 (assistant-ui Thread + Composer), shell layout
- **Composition:** `<ThreadPrimitive.Root>` wrapping `<HeaderBar>` + `<ThreadPrimitive.Viewport>` + `<ThreadPrimitive.ViewportFooter>`-wrapped `<Composer>`. Root is `flex h-full flex-col items-stretch bg-background text-foreground`.
- **Layout:** `max-w-3xl mx-auto` content column inside a full-bleed background.
- **State:** holds the `AssistantChatTransport` runtime (Section "Recommended Project Structure" in AI-SPEC) — `runtime.ts` is a sibling import.

### 2. HeaderBar — `src/ui/components/HeaderBar.tsx`

- **Implements:** D-15 recompile button placement + D-16 idle status content + D-17 in-flight status
- **Composition:** `<header>` element, `h-14 border-b border-border bg-muted px-6 flex items-center justify-between`. Left: `<h1 class="text-2xl font-medium">Business Strategy Planner</h1>`. Right: `<RecompileStatus />` followed by `<RecompileButton />` with `gap-3` between them.
- **Behavior:** the status pill polls `GET /recompile/status` (or reads from a shared SSE subscription if implementer prefers — AI-SPEC leaves this open) on a 5s interval *only when not in-flight*; while in-flight, status is pushed by the same SSE channel that the recompile button kicked off.
- **Sticky:** `sticky top-0 z-10` so it stays visible during long chat scrolls.

### 3. RecompileButton — `src/ui/components/RecompileButton.tsx`

- **Implements:** UI-06 + D-15
- **Composition:** shadcn `<Button variant="default">` (which renders the `bg-primary` accent treatment). Width: `auto`, `h-9` (36px — 4px shy of the 56px header height to allow the bar's vertical padding to breathe). Icon `RefreshCwIcon size-4` to the left of label `Recompile`, `gap-2` between.
- **Disabled rules:** disabled (`disabled:opacity-50 disabled:pointer-events-none`) while a recompile is in flight; not disabled when 0 claims are unwritten (the user can still recompile to re-run determinism — Phase 1 D-18 short-circuits via hash).
- **Slash-command parity:** the `/recompile` composer slash command (D-15) calls the same backend route. The composer parses `/recompile` *before* sending the message to `query()` and routes to `POST /recompile` instead. No visible UI for the slash command — it's a power-user shortcut.

### 4. RecompileStatus pill — `src/ui/components/RecompileStatus.tsx`

- **Implements:** D-16 idle, D-17 in-flight
- **Composition:** `<span>` with `h-7 px-3 inline-flex items-center gap-2 rounded-full bg-muted text-xs text-muted-foreground border border-border`. Idle: text per Copywriting "Recompile status pill — idle". In-flight: text per "Recompile status pill — in-flight"; spinner `Loader2Icon size-4 animate-spin` prefix replaces the bullet separator.
- **Color:** muted-secondary (30% palette), not accent. The pill is *informational*, not actionable; its accent-colored neighbor button is the actionable surface.
- **Reading order:** "Last compiled" semantic comes first (left), dirty-claim count right of bullet — so a quick glance reads "X minutes ago" before drilling into "and N pending."

### 5. ToolTrace — `src/ui/components/ToolTrace.tsx`

- **Implements:** UI-03 + D-08 + D-11 + D-12
- **Composition (collapsed):** `<button>` element rendering exactly the copywriting template (`▸ N tool calls (research, M tavily_extract, K onebrain_write_claim)`). The button is `w-fit text-xs text-muted-foreground hover:text-foreground transition-colors py-1`. Sits below the assistant message body with `mt-2` spacing.
- **Composition (expanded):** chevron rotates to down (`rotate-90`); a `<ul>` renders below with each tool call on its own row at `text-xs leading-5`. Row format per D-12: `<tool-name>(<args-summary>) → <result-summary>`. The tool-name span uses `font-mono text-foreground` (this is the one place mono is allowed — see Typography note); args/result summary is `text-muted-foreground`. Rows are `py-1 px-2 hover:bg-muted rounded-md` to give a subtle hover affordance for "show raw" (which is implementer's choice per CONTEXT §"Claude's Discretion" — Phase 2 may ship this, may defer).
- **Tool-event source:** SDK `onToolCall` / `onToolResult` hooks stream into a `toolTraceSink.emit(e)` (AI-SPEC §"Entry Point Pattern"); the SSE bridge forwards them to the browser, where the runtime stores per-message tool events keyed by message ID. Component renders from runtime state; no direct fetch.
- **Two-channel transparency (D-08).** The one-line *prose* intent ("Researching Acme's pricing model on the web…") is a *separate* assistant message rendered above the eventual reply, not inside the ToolTrace. The ToolTrace sits below the *reply* message, not below the intent message — this preserves the two-channel split (prose → ambient awareness; trace → verification).

### 6. WikiCitation — `src/ui/components/WikiCitation.tsx`

- **Implements:** UI-04 + D-13 + D-14
- **Composition:** a `<div>` block, `bg-muted border border-border rounded-lg p-4 my-4` (renders inline inside an assistant message above the prose answer per D-13 "surfaces an inline excerpt above its prose answer"). Header: `<small class="text-xs text-muted-foreground">From the wiki: <topic-slug>.md</small>`. Body: the ~200-word excerpt (D-14) rendered as Markdown via assistant-ui's `<MessagePrimitive.Content>` markdown renderer (so tables/callouts/etc. carry through). Footer: a `flex items-center gap-2 mt-3` row with the "Open in Obsidian →" button (`<Button size="sm" variant="default">` — accent-treated per the reserved-for list) and a fallback `Copy path` button (silent fallback per D-14, secondary `<Button size="sm" variant="outline">`).
- **Excerpt cap behavior:** if the source markdown is > 200 words, truncate at the closest paragraph boundary ≤ 200 words and append a `text-xs text-muted-foreground italic` line `(excerpt — full page in Obsidian)`. Never mid-sentence-truncate.
- **`obsidian://` URL construction:** `obsidian://open?vault=<vault-name>&file=<encoded-path>`. Vault name from a config value (default = directory basename `vault`). File path is the relative path from the vault root, URL-encoded. This is the wiki-citation block's only side-effect-bearing affordance; on click, the browser hands off to the OS-registered `obsidian://` handler. Detection of "scheme not registered" is fire-and-forget — assistant-ui doesn't get a callback if the user's OS has no handler. The fallback "Copy path" button is *always rendered* alongside the deeplink (not a fallback shown only on failure) — it's a low-cost belt-and-braces affordance.

## Interaction Contracts

### IC-1 — Streaming chat (UI-02)

- Composer submit → POST /chat → SSE stream → assistant-ui's `AssistantChatTransport` consumes UIMessageChunks → message viewport renders incrementally token-by-token.
- **Visual contract:** assistant message bubble appears immediately on first chunk (not after the full reply) with a thin blinking cursor at the end of streamed text. Cursor disappears on stream end.
- **Latency target:** ≤ 5s first token (AI-SPEC §"Latency budget"). If first token > 8s, render a "Thinking…" prose message in the bubble.

### IC-2 — Sub-agent invocation narration (D-08)

- Coordinator emits a one-line intent message *before* invoking a sub-agent.
- **Visual contract:** the intent message renders as a normal assistant message (`bg-muted`, `text-foreground`, body type) with no special treatment — it reads as conversational.
- The follow-up reply (the actual answer) renders as a *separate* assistant message below it. The ToolTrace component attaches to the *reply*, not the intent.

### IC-3 — Tool-trace expand/collapse (D-11)

- Default: collapsed (chevron right, single-line summary).
- Click anywhere on the summary row → expand (chevron down, list of rows).
- Click again → collapse.
- Per-message state: each assistant message's trace state is independent and ephemeral (resets on page refresh). No persistence.

### IC-4 — Recompile lifecycle (D-15..D-18)

- User clicks Recompile button OR types `/recompile` in composer.
- Button → `disabled` + label flips to `Compiling…` + spinner. Status pill flips to in-flight content. SSE subscription begins delivering progress events from `POST /recompile`.
- On completion: button flips back to idle. Status pill flips to `Last compiled: now • <N> claims unwritten` (`<N>` should be 0 immediately after a successful recompile, per D-16's dirty-count formula). Chat receives a system message per D-18.
- On error: button flips back to idle but in `outline` variant + destructive text-color. Status pill copy: `Last compile failed: <HH:MM> — see Obsidian log`. Chat receives a destructive-color system message naming the error.

### IC-5 — Slash-command parsing in composer

- Composer detects a leading `/` on the first character of the input.
- For Phase 2 the only known command is `/recompile`. Other `/foo` strings are sent as normal chat messages (the coordinator may interpret them, but the composer does not preempt).
- On `/recompile` (with optional whitespace and no further args): composer DOES NOT submit to `/chat`; it submits to `/recompile` and clears the composer. UI-side, this is indistinguishable from clicking the Recompile button.

### IC-6 — Empty-state rendering

- When `thread.isEmpty` (assistant-ui state), the viewport renders an `<AuiIf>` block centered (`flex grow flex-col items-center justify-center px-8 py-16 gap-6`) containing:
  - The empty-state heading (`text-lg font-medium`)
  - The empty-state body (`text-sm text-muted-foreground max-w-md text-center`)
- No avatar / illustration in Phase 2. Keep visual mass low so the composer remains the obvious next action.

### IC-7 — Inline claim-citation rendering (D-09)

- The coordinator's prose includes `[[claim:01J9X…]]` literal tokens.
- The assistant-ui markdown renderer is configured with a custom inline-token rule that maps `[[claim:<id>]]` to a `<button>` with `text-primary underline-offset-2 hover:underline text-xs font-medium`.
- Click → for Phase 2, opens a tooltip (radix `<Tooltip>`) with the live OneBrain row's `text + confidence + status`. (No modal; no separate panel. Tooltip is enough for Phase 2 verification.)
- Phase 5's UI-05 will replace this tooltip with a richer inline confidence-badge element; for Phase 2, prose + tooltip is the substitute.

## Accessibility

- **Color contrast:** the chosen tokens (`text-foreground` on `bg-background` = `#0A0A0A` on `#FFFFFF`, ratio 19.5:1; `text-muted-foreground` on `bg-muted` = `#71717A` on `#F4F4F5`, ratio 4.6:1) all clear WCAG AA for body text. The accent button (`text-primary-foreground` on `bg-primary` = `#FAFAFA` on `#18181B`, ratio 17.4:1) clears AAA.
- **Focus visibility:** Tailwind v4's default `focus-visible:ring-2 focus-visible:ring-ring focus-visible:outline-none` is mandatory on every interactive element (Button, ToolTrace summary, claim-citation links, composer input). Do not strip outlines without replacement.
- **Keyboard operability:** all interactive elements are reachable by Tab in source order. Composer is the natural last stop. Recompile button has explicit `aria-label="Recompile vault"` because the icon + "Recompile" label still benefits from explicit labeling for screen readers.
- **Live regions:** the recompile status pill updates inside an `aria-live="polite"` region so screen readers announce state transitions. The chat viewport itself is `aria-live="polite"` per assistant-ui defaults.
