# Wiki commit policy — committed to git with manual regen

The `wiki/` folder is **checked into git** and regenerated by an explicit, deterministic `pnpm wiki:regen` command. There is no auto-regen wired into Application use-cases or pre-commit hooks.

## Considered Options

- **A — Committed + manual regen (chosen).** `wiki/` checked in; `pnpm wiki:regen` reads OpenBrain → writes markdown; user (or agent) decides when to snapshot.
- **B — Gitignored + always regenerate.** No git history at file level; always reflects live OpenBrain state.
- **C — Committed + auto-regen on every projection-relevant write.** Application use-cases that mutate OpenBrain trigger wiki regen for affected pages as part of the same logical operation.

## Why A

- **Honours the OpenBrain-as-source-of-truth + wiki-as-projection metaphor.** A projection is a snapshot, deliberately. Auto-regen (option C) tries to make the snapshot "always live," which conflicts with the projection metaphor and produces noisy git diffs on every Hypothesis transition.
- **Simplest mental model.** One rule: "run `wiki:regen` when you want a fresh snapshot." The user (or agent) decides when freshness matters — typically before opening Obsidian or before sharing/exporting.
- **Free backup + audit trail.** Git history of `wiki/` is genuinely useful — "what did my SWOT look like at the start of Q2" becomes a `git log wiki/swot/` query. Committed files give this for free without standing up a separate snapshot system. Combined with `pg_dump` cron on OpenBrain, the wiki gets a *second* tier of backup as a side effect.
- **Clean layer boundaries.** Application use-cases don't know the wiki exists. Wiki regen is a separate `packages/external/wiki/` concern that reads from OpenBrain and writes filesystem. No coupling.

## Why not B

- Loses the "wiki is git-versioned" backup pattern already implied in [personal-use-tradeoffs.md](../principles/personal-use-tradeoffs.md#L37).
- Loses the time-machine "what did the wiki look like on 2026-04-15" ability — replicating it would require restoring a `pg_dump` and running regen, which is heavier than `git log wiki/`.

## Why not C

- Every Hypothesis state-machine transition would generate a wiki diff in the next commit. Commit diffs become noisy; semantic intent of a code change gets buried under wiki re-render churn.
- Couples Application use-cases to the wiki adapter — exactly what the Application/External port-and-adapter pattern in [architecture.md](../principles/architecture.md) exists to prevent.
- The "wiki is always live" property C aims for can be approximated more cheaply via a freshness badge in the chat UI (see Consequences).

## Determinism requirement

`pnpm wiki:regen` MUST be deterministic — same OpenBrain state ⇒ identical wiki bytes — otherwise git diffs show spurious changes that hide real ones. Concretely:

- Stable ORDER BY on every Postgres read that drives a wiki page.
- Stable YAML frontmatter key order (alphabetised; or a fixed list in code).
- No timestamps embedded in page bodies (timestamps belong in frontmatter, where they reflect data state, not generation time).
- No randomness, no hashes-of-process-state.

This is a hard rule on the wiki renderer; the test for "regen is deterministic" is "running it twice in a row produces no git diff."

## Consequences

- **Freshness is the user's responsibility.** Mitigation: the chat UI surfaces a "wiki is N OpenBrain-events behind — regen?" badge, computed by comparing per-page content hashes vs current OpenBrain projection. (Specifics deferred to the implementation issue.)
- **Pre-commit hook does NOT regen.** Regen stays explicit. Forcing regen on every commit makes commits slow and pollutes diffs.
- **Full regen for v0.** Per-page incremental regen is a follow-up optimisation if regen-time becomes painful (unlikely below ~1000 pages).
- **Obsidian-compatible:** `wiki/` is the top-level vault. Frontmatter, `[[wikilinks]]`, no special characters that confuse Obsidian. (Already in [memory-architecture.md](../principles/memory-architecture.md#L41).)
